# When Reward Signals Disagree

**Aakarsh Karit | February 2026**

In January 2025, a team at Shanghai Jiao Tong University trained a language model to answer medical questions. The setup was straightforward. They used reinforcement learning with two reward signals: one checked whether the model's answer was formatted correctly (wrapped in the right XML tags), and another checked whether the answer was actually right. Both rewards were summed together. Both worked fine on their own.

What happened next was not fine. The model learned, within a few hundred training steps, that producing a correctly formatted but trivially short answer was the path of least resistance. A well-tagged one-word response scores high on format compliance and occasionally guesses the right answer. The team called this the Direct Answer Hacker. The model had found a crack in the seam between two rewards and driven a wedge through it.

This is not an isolated case. DeepSeek-R1, one of the most capable open reasoning models, exhibited similar behavior during training. The model learned to produce long chains of reasoning not because they helped it think, but because the length correlated with higher scores from the language model judge evaluating quality. A chess-playing RL system trained by Bondarenko et al. found specification gaps between its move-legality checker and its win-condition evaluator, producing games that satisfied every individual reward while playing nonsensical chess. In each case the same pattern appears. Every reward component is doing exactly what it was told to do. The failure lives in how the components are combined.

This is a composition problem.

Modern RL training pipelines for language models do not use a single reward. They compose many: code execution results, language model judge scores across multiple rubric dimensions, safety classifier outputs, format validators, length penalties, and KL divergence regularization. A typical production training run at a frontier lab might compose ten or more heterogeneous signals into the single scalar that drives gradient updates. The function that performs this composition is almost always an ad-hoc weighted sum with hand-tuned coefficients. Sometimes with clipping. Sometimes with normalization. Never with formal security guarantees.

The field treats reward composition as a hyperparameter tuning problem. We think it is a security problem.

The distinction matters. A hyperparameter problem says: the weights are not quite right, tune them better. A security problem says: there are structural properties that the composition function must satisfy regardless of the weights, and violating these properties creates exploitable attack surfaces. A sufficiently capable model trained against a composition function that lacks these properties will find and exploit the gaps. Not because the model is malicious, but because gradient descent is an optimizer and optimizers find shortcuts.

Computer security dealt with a structurally identical problem twenty years ago. Mark Miller's 2006 dissertation, Robust Composition, addressed exactly this question for software: when you compose separately written programs so they may cooperate, how do you prevent them from destructively interfering in ways nobody anticipated? His answer was object-capability security, a set of formal principles including the Principle of Least Authority (give each component only the information and permissions it needs), authority attenuation (no component can amplify its influence beyond what it was granted), and membrane patterns (hierarchical boundaries that control what flows between components).

These principles are now foundational in systems security. They underpin the seL4 verified microkernel, the Agoric smart contract platform, and the security architecture of several major web browsers. But nobody has applied them to RL reward composition. The ML community and the security community simply have not overlapped on this particular problem.

We are building a composition layer called rewardcap that enforces these security properties for reward signals in RL training. It is not a new RL algorithm. It does not change how individual reward components work. It sits between the reward computation and the optimizer, enforcing six formal properties derived from Miller's framework: authority attenuation (no reward can amplify its influence beyond its bound), information isolation (each reward sees only what it needs), independence guarantees (rewards meant to measure different things actually do), revocable authority (rewards that degrade over training are detected and handled), hierarchical gating (format compliance is a prerequisite, not an optimization target), and bounded composition (the aggregate reward is well-behaved).

The Direct Answer Hacker fails under this framework because format compliance becomes a gate rather than an additive score. Once format is satisfied, it contributes zero gradient. The model cannot trade format compliance against correctness because they operate in different layers of the composition hierarchy. The DeepSeek-style length exploitation fails because the language model judge operates under information isolation and authority attenuation, preventing its scoring preferences from dominating the gradient.

We have preliminary results showing these properties working in controlled settings: simulated composition scenarios where ad-hoc weighted sums produce exactly the documented failure modes and where principled composition prevents them. The core framework is implemented. The formal properties are specified. We are now running validation experiments on small-scale RL training.

This project is funded in part by the Foresight Institute, whose research program has supported foundational work in object-capability security (Mark Miller) and formally verified systems (the seL4 team). We see rewardcap as extending that tradition to a new domain. The reward composition function in an RL training pipeline is a trust boundary. It is the place where multiple independent evaluation signals merge into the single number that shapes model behavior. Securing that boundary is the same kind of problem as securing a system call interface or a smart contract execution environment. The tools exist. They just have not been applied here yet.
